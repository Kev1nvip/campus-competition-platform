## 一、项目概述

### 1.1 项目背景

高校学术竞赛管理普遍存在以下问题：

```
信息不对称：
  竞赛信息分散在各处，学生不知道有哪些竞赛适合自己
  不知道哪个老师在带哪个竞赛，老师也不知道哪些学生有意向

组队混乱：
  团队赛靠微信私信找队友，没有统一平台，全靠人脉
  队伍信息不透明，导师难以掌握全局

流程不透明：
  报名到哪一步了？审核了没有？学生只能被动等通知
  管理员靠Excel统计数据，效率低

获奖记录分散：
  获奖信息靠学生自己记，学院没有统一的竞赛获奖数据库
  老师带队成绩无法横向比较，缺乏参考依据
```

### 1.2 项目定位

```
一站式校园学术竞赛管理平台

核心价值：
  对学生：一个平台看所有竞赛，找队友找老师有据可查
  对老师：统一管理带队事务，历史成绩可展示
  对学校：竞赛数据沉淀，获奖记录可追溯，管理效率提升
```

### 1.3 目标用户

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| 学生 | 在校本科生/研究生 | 发现竞赛、组队报名、查看进度 |
| 老师 | 指导教师 | 发布招募、审核学生、管理带队 |
| 管理员 | 教务处/团委管理员 | 审核报名、统计数据、管理用户 |

### 1.4 项目范围

**本期实现（v1.0）**

```
✅ 用户注册登录与权限管理
✅ 竞赛信息发布与管理
✅ 个人赛报名全流程
✅ 团队赛组队全流程
✅ 招募帖发布与申请
✅ 三级审核流程（老师确认→管理员审核）
✅ 消息通知
✅ 获奖记录管理
✅ 数据统计
```

**本期不做（后续迭代）**

```
❌ 移动端App
❌ 竞赛官方报名对接（蓝桥杯等有自己的系统）
❌ 在线支付报名费
❌ 视频/直播功能
```

## 三、功能需求

### 3.1 用户模块

#### 3.1.1 注册

**功能描述**

用户填写信息完成注册，角色分为学生和老师，管理员账号由系统初始化。

**字段说明**

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|----------|------|
| username | String | 是 | 4-20位，字母数字下划线 | 全局唯一 |
| password | String | 是 | 6-20位 | 加密存储 |
| realName | String | 是 | 2-10位汉字 | 真实姓名 |
| role | Enum | 是 | STUDENT/TEACHER | 注册后不可修改 |
| phone | String | 否 | 11位手机号 | - |
| email | String | 否 | 邮箱格式 | - |
| studentNo | String | 学生必填 | 纯数字 | 学号 |
| department | String | 是 | - | 院系 |
| title | String | 老师必填 | - | 职称，如讲师/副教授 |

**业务规则**

```
1. username 全局唯一，重复时提示"用户名已存在"
2. studentNo 全局唯一，重复时提示"学号已被注册"
3. 注册成功后直接登录，无需邮件验证
4. 管理员账号不开放注册，由初始化脚本创建
```

**异常处理**

| 异常场景 | 提示信息 |
|----------|----------|
| 用户名已存在 | 用户名已被使用，请更换 |
| 学号已注册 | 该学号已被注册，请确认 |
| 参数格式错误 | 具体字段的校验提示 |

---

#### 3.1.2 登录

**功能描述**

用户通过用户名和密码登录，返回JWT Token，前端存储后用于后续请求鉴权。

**业务规则**

```
1. 用户名或密码错误：提示"用户名或密码错误"，不区分具体哪个错
2. 账号被禁用：提示"账号已被禁用，请联系管理员"
3. Token有效期：24小时
4. Token过期后：前端跳转登录页，清除本地存储
```

#### 3.1.3 个人主页

**学生个人主页展示内容**

```
基本信息：姓名、学号、院系、联系方式
参赛记录：
  进行中的竞赛（含状态）
  历史参赛记录
获奖记录：
  所有已确认的获奖记录
  奖项等级、竞赛名称、获奖时间

**老师个人主页展示内容**

基本信息：姓名、职称、院系、联系方式

当前带队：
  各竞赛当前已带队数 / 最多可带队数
  （学生选老师时参考）
```

### 3.2 竞赛模块

#### 3.2.1 发布竞赛

**功能描述**

管理员或老师发布竞赛信息，学生可查看并报名。

**字段说明**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | String | 是 | 竞赛名称，最长128字 |
| type | Enum | 是 | INDIVIDUAL个人赛 / TEAM团队赛 |
| organizer | String | 是 | 主办方 |
| requirement | String | 否 | 参赛要求，如专业/年级限制 |
| signupStart | DateTime | 是 | 报名开始时间 |
| signupEnd | DateTime | 是 | 报名截止时间 |
| competitionStart | DateTime | 否 | 比赛开始时间 |
| competitionEnd | DateTime | 否 | 比赛结束时间 |
| hasQuota | Boolean | 是 | 是否有名额限制 |
| maxQuota | Integer | 条件必填 | hasQuota为true时必填，最小值1 |
| minTeamSize | Integer | 团队赛必填 | 队伍最少人数 |
| maxTeamSize | Integer | 团队赛必填 | 队伍最多人数 |
| maxTeachQuota | Integer | 否 | 每位老师最多带队数，不填表示不限制 |
| description | String | 否 | 竞赛详情 |
| attachmentUrl | String | 否 | 附件，竞赛通知文件 |

**竞赛状态流转**

```
UPCOMING（未开始）
    ↓ 到达signupStart时间，自动切换
SIGNING（报名中）
    ↓ 到达signupEnd时间，自动切换
CLOSED（报名截止）
    ↓ 到达competitionStart时间，自动切换
ONGOING（进行中）
    ↓ 到达competitionEnd时间，自动切换
FINISHED（已结束）

手动操作：
  发布人可以随时将竞赛下架（变为OFFLINE状态）
  下架后学生不可见，不可报名
```

**业务规则**

```
1. signupEnd 必须晚于 signupStart
2. competitionStart 必须晚于 signupEnd
3. 团队赛 minTeamSize 必须 ≥ 2
4. 团队赛 minTeamSize 必须 ≤ maxTeamSize
5. 竞赛发布后状态自动根据时间流转
6. 只有发布人或管理员可以编辑/下架竞赛
```

### 3.3 报名模块

#### 3.3.1 个人赛报名流程

**流程说明**

```
路径A（学生主动找老师）：
  Step1：学生选择竞赛，点击报名
  Step2：选择指导老师（可查看老师历史带队记录）
  Step3：填写报名信息，提交指导申请
  Step4：老师同意/拒绝指导申请
         拒绝：学生收到通知，可重新选择其他老师
         同意：进入Step5
  Step5：学生提交管理员审核
  Step6：管理员审核
         通过：报名正式生效
         驳回：退回给老师，老师通知学生修改后重新提交

路径B（老师主动招募）：
  Step1：老师发布招募帖（关联竞赛）
  Step2：学生申请加入招募帖
  Step3：老师审核学生申请，同意加入
  → 进入路径A的Step5
```

**报名信息字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| competitionId | Long | 是 | 竞赛ID |
| teacherId | Long | 是 | 指导老师ID |
| motivation | String | 否 | 参赛动机 |
| introduction | String | 否 | 个人简介 |

**业务规则**

```
1. 同一学生可以同时报名多个竞赛（无限制）
2. 竞赛在报名截止后不可新增报名
3. 有名额限制时，达到上限后不可报名
4. 同一竞赛同一学生只能有一条有效报名记录
5. 报名被驳回后可以修改重新提交
6. 报名生效后不可取消（需联系管理员）
```

**老师带队限制**

```
竞赛配置了maxTeachQuota时：
  学生选择老师时，显示该老师当前带队数/上限
  已满时：该老师显示"名额已满"，学生不可选择
  未满时：正常发送申请

并发控制：
  使用Redis计数器记录老师在该竞赛的带队数
  老师同意申请时：Redis计数器+1，同时数据库乐观锁校验
  并发冲突时：返回失败，提示"该老师名额已满"
```

#### 3.3.2 团队赛报名流程

**流程说明**

```
路径A（老师主导）：
  Step1：老师发布招募帖，关联竞赛，填写招募人数和要求
  Step2：学生申请加入招募帖
  Step3：老师审核申请，同意/拒绝
         人数满足竞赛要求后，老师确认组队完成
  → 进入合并节点

路径B（学生主导）：
  Step1：学生创建队伍，成为队长
  Step2：队长选择指导老师，发送带队申请
  Step3：老师同意带队（检查带队数量上限）
  Step4：老师同意后，队长发布组队招募帖或直接邀请队友
         方式1：发布招募帖，等待其他学生申请
         方式2：搜索学号/姓名，直接发送邀请
  Step5：队友确认加入
         人数满足竞赛最少人数要求后，队长可提交审核

→ 合并节点（两条路统一进入）：
  Step6：队长提交管理员审核（只有队长能操作）
  Step7：管理员审核
         通过：报名正式生效
         驳回：退回给老师，老师通知队长修改后重新提交
  Step8：队长修改后重新提交
```

**队伍信息字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| competitionId | Long | 是 | 关联竞赛 |
| teamName | String | 是 | 队伍名称，最长64字 |
| teacherId | Long | 是 | 指导老师ID |

**业务规则**

```
队伍相关：
  1. 同一竞赛，一个学生只能加入一支队伍
  2. 同一竞赛，一个学生可以同时参加其他竞赛
  3. 只有队长能提交审核、解散队伍、踢出成员
  4. 队员可以主动退出队伍（提交审核前）
  5. 队伍提交审核后，成员不可退出，队长不可踢人

老师相关：
  6. 路径B中，必须先获得老师同意才能发组队招募帖
  7. 老师同意带队后，带队计数器+1
  8. 老师拒绝带队，队长需重新选择其他老师

人数相关：
  9. 提交审核时，队伍人数必须满足竞赛minTeamSize要求
  10. 队伍人数达到maxTeamSize后，招募帖自动关闭
```

### 3.4 招募帖模块

#### 3.4.1 老师招募帖

**功能描述**

老师发布招募帖，面向学生招募参赛成员。个人赛和团队赛均可发布。

**字段说明**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| competitionId | Long | 是 | 关联竞赛 |
| recruitCount | Integer | 是 | 招募人数 |
| requirement | String | 否 | 对学生的要求描述 |
| deadline | DateTime | 否 | 招募截止时间 |

**业务规则**

```
1. 老师发布招募帖前检查带队数量上限
   已达上限：不允许发布
2. 人数达到recruitCount后，招募帖自动关闭
3. 竞赛报名截止后，自动关闭所有关联招募帖
4. 老师可手动关闭招募帖
5. 同一老师同一竞赛可发布多个招募帖（不同需求）
```

**学生申请字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| recruitmentId | Long | 是 | 招募帖ID |
| introduction | String | 否 | 个人简介 |
| motivation | String | 否 | 申请理由 |

---

#### 3.4.2 学生组队招募帖

**功能描述**

团队赛中，队长在获得老师同意带队后，发布招募帖寻找队友。

**字段说明**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| teamId | Long | 是 | 关联队伍 |
| recruitCount | Integer | 是 | 还需要几人 |
| requirement | String | 否 | 对队友的要求 |
| deadline | DateTime | 否 | 招募截止时间 |

**业务规则**

```
1. 前提：该队伍已有老师确认带队
2. 队伍人数达到maxTeamSize后，招募帖自动关闭
3. 竞赛报名截止后，自动关闭
4. 只有队长可以发布/关闭组队招募帖
```

### 3.5 审核模块

#### 3.5.1 管理员审核

**触发时机**

```
个人赛：老师同意指导后，学生手动提交
团队赛：老师同意带队且人数满足要求后，队长手动提交
```

**审核内容**

```
管理员查看：
  参赛人员基本信息（姓名、学号、院系）
  指导老师信息
  竞赛信息
  是否满足竞赛参赛要求（专业/年级）
  材料是否完整

审核操作：
  通过：报名正式生效，通知学生/队长
  驳回：填写驳回原因，退回给老师
```

**驳回后流程**

```
管理员驳回
    ↓
老师收到通知，查看驳回原因
    ↓
老师通知学生/队长修改
    ↓
学生/队长修改后重新提交
    ↓
再次进入管理员审核队列
```

**审核状态**

| 状态 | 说明 |
|------|------|
| PENDING | 待管理员审核 |
| APPROVED | 审核通过，报名生效 |
| REJECTED | 审核驳回，已退回 |
| RESUBMITTED | 修改后重新提交 |

### 3.6 获奖记录模块

#### 3.6.1 提交获奖记录

**触发时机**

竞赛状态变为FINISHED后，学生可上传获奖记录。

**字段说明**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| signupId | Long | 是 | 关联的报名记录 |
| awardLevel | Enum | 是 | 奖项等级（见下表）|
| awardName | String | 是 | 奖项名称，如"全国一等奖" |
| certificateUrl | String | 是 | 获奖证书图片 |
| awardDate | Date | 是 | 获奖日期 |

**奖项等级枚举**

| 枚举值 | 说明 |
|--------|------|
| NATIONAL_FIRST | 国家级一等奖 |
| NATIONAL_SECOND | 国家级二等奖 |
| NATIONAL_THIRD | 国家级三等奖 |
| PROVINCIAL_FIRST | 省级一等奖 |
| PROVINCIAL_SECOND | 省级二等奖 |
| PROVINCIAL_THIRD | 省级三等奖 |
| OTHER | 其他 |

**业务规则**

```
1. 只有报名已生效（APPROVED）的学生才能提交获奖记录
2. 团队赛：队长或任意队员均可提交，关联整支队伍和老师
3. 一条报名记录只能提交一条获奖记录
4. 提交后等待管理员审核
5. 驳回后可修改重新提交
```

#### 3.6.2 管理员审核获奖记录

```
审核内容：
  核实证书图片真实性
  确认奖项等级与证书一致
  确认获奖人员与报名信息一致

审核通过：
  获奖记录正式入库
  展示在学生个人主页
  展示在老师主页带队成绩中

审核驳回：

### 3.9 AI竞赛推荐模块

#### 3.9.1 功能描述

```
学生输入自己的专业、技能、感兴趣方向
系统基于RAG技术检索竞赛知识库
返回适合该学生的竞赛推荐列表
每条推荐附带推荐理由和知识库来源引用
```

#### 3.9.2 知识库来源

```
知识库内容：
  收集以下竞赛的官方介绍文档，整理为txt格式入库

  计划收录竞赛（5-10个）：
  1. 蓝桥杯全国软件和信息技术专业人才大赛
  2. 全国大学生数学建模竞赛
  3. ACM-ICPC国际大学生程序设计竞赛
  4. 中国大学生计算机设计大赛
  5. 全国大学生电子设计竞赛
  6. 华为软件精英挑战赛
  7. 全国大学生数据分析大赛
  8. 互联网+大学生创新创业大赛

每个竞赛的txt文档包含：
  竞赛简介
  主办方
  参赛对象（专业/年级要求）
  竞赛形式（个人/团队，人数要求）
  历届赛题方向
  备赛建议
  往届获奖情况参考
```

#### 3.9.3 技术实现

```
框架：LangChain4j
向量数据库：PGVector（PostgreSQL插件，无需单独部署）
LLM：智谱AI GLM / 通义千问（调用API）
Embedding模型：对应平台的Embedding API

RAG链路：
  文档入库时：
    txt文档 → 按段落分块（Chunk）
    → Embedding向量化
    → 存入PGVector

  用户提问时：
    用户输入 → Embedding向量化
    → PGVector相似度检索，召回Top5相关片段
    → 构建Prompt（含召回内容）
    → LLM生成推荐结果
    → 返回推荐列表+来源引用
```

#### 3.9.4 接口说明

```
POST /api/v1/ai/recommend

入参：
  direction：String，学生的方向描述
  例："我是计算机专业大三，擅长Java后端开发，想提升简历"

出参：
  recommendations：推荐竞赛列表
    - competitionName：竞赛名称
    - reason：推荐理由
    - source：知识库来源文档名
    - matchScore：匹配度（高/中/低）
```

#### 3.9.5 演示说明

```
答辩演示步骤：
  Step1：输入"我是软件工程大三，擅长算法和数据结构"
  Step2：系统返回推荐（蓝桥杯、ACM等）
  Step3：展示每条推荐的来源引用
  Step4：说明RAG相比直接调用LLM的优势：
         "答案基于平台知识库，有据可查，不会乱编"
```

## 六、验收标准

### 6.1 功能验收

**个人赛完整流程**

```
□ 学生可以查看竞赛列表并筛选个人赛
□ 学生可以查看老师主页，了解带队历史
□ 学生提交报名申请，老师收到通知
□ 老师同意后，学生提交管理员审核
□ 管理员审核通过，双方收到通知
□ 管理员驳回，老师收到通知并能通知学生重提
□ 竞赛结束后，学生可上传获奖证书
□ 管理员审核通过，获奖记录展示在个人主页和老师主页
```

**团队赛完整流程**

```
□ 老师发布招募帖，学生可以申请加入
□ 老师审核申请，同意/拒绝通知到位
□ 学生创建队伍，选择老师发送带队申请

□ 队友邀请/申请加入，通知到位
□ 人数满足要求，队长提交管理员审核
□ 管理员审核驳回，老师和队长均收到通知
□ 队长修改后重新提交，流程正常推进
```

```
□ 有名额限制的竞赛，100并发报名无超报
